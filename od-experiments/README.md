# od-experiments — DROID reconstruction → satellite map + 3D object detections

A from-scratch web app that takes a VINGS object-detection run (gauge-free DROID
2D-Gaussian PLY + per-object detections + tracker poses) and:

1. **renders the reconstruction** as a top-down point cloud in the browser (three.js),
2. **projects it into a real satellite map** (Esri World Imagery), GPS-anchored so the
   strip lands at its true location, heading and metric scale — fixing the "warped /
   wrong way around" DROID frame,
3. **draws the detected vehicles as per-class glTF models** (`car/van/truck/bus`) at
   their detected ground positions, with real vehicle dimensions and heading.

Everything is validated headless via Playwright screenshots compared against the
satellite imagery and the run's overlay PNGs.

## Layout
```
app/                viewer (static; serve over http, fetch() won't work on file://)
  index.html        HUD + importmap
  main.js           three.js scene: satellite plane + point cloud + GLB objects
  vendor/           three.js r160 (module + OrbitControls/GLTFLoader) vendored local
  assets/           generated by tools/preprocess.py (+ models/*.glb)
tools/
  preprocess.py     PLY+GPS+poses -> cloud_pos.f32 / cloud_col.u8 / objects.json /
                    scene.json / satellite.png   (the whole geo pipeline)
  align.py          builds the cloud orthophoto + tries cross-modal registration
                    (kept for diagnostics; auto-refine is unreliable here -> global)
  shot.py           headless render+screenshot (validated swiftshader recipe)
  serve.py          local http server for interactive use
shots/              screenshots (validation + hero frames)
```

## Run
```bash
conda activate vings
python tools/preprocess.py            # regenerate assets (global GPS-anchored frame)
python tools/serve.py                 # then open http://127.0.0.1:8000/index.html
```
Headless screenshot (note the ulimit — see below):
```bash
ulimit -s 1024
python tools/shot.py top out.png "copac=0.9&psize=1.7&obj=1&path=1"
```
URL params: `view=top|persp|obl`, `sat/cloud/obj/path=0|1`, `psize`, `copac`,
`zoom`, `pe`/`pn` (pan east/north metres).

## How the geo-projection works
The flight GPS (`rtk.csv`) is a **near-perfect straight line** → Umeyama rotation is
degenerate (can't pin roll/up). So the world frame is built from:
* **up**     = `-mean(camera forward)` (nadir cam looks down),
* **heading**= DROID start→end **chord**, rotated onto the GPS start→end chord,
* **scale**  = GPS chord length / DROID along-track span,
* **position**= path-centroid match.

Then a degree-3 along-track **vertical de-trend** removes the smooth DROID altitude
drift (≈67 m over 308 m) so the ground is level and objects sit on it. A per-keyframe
local-Sim3 unwarp exists (`UNWARP=1`) but bow-ties this near-straight flight, so the
global frame is the default.

### Gotchas baked in
* **Mirror bug**: the DROID basis must be right-handed (`right = fwd × up`); using
  `up × fwd` makes the whole transform a reflection and the cloud comes out mirrored.
* **Headless Chrome under strict overcommit** (`vm.overcommit_memory=2`): launch with
  `--single-process --no-zygote --use-gl=swiftshader` and `ulimit -s 1024`, else
  Chrome dies with `pthread_create EAGAIN` / `spawn ENOMEM`. (See `tools/shot.py`.)
* **Auto cloud↔satellite refinement** (ECC/ORB/phase-corr in `align.py`) does not
  converge — different capture date, sparse 28 % ortho coverage, reconstruction noise.
  Alignment is therefore the physics-grounded global frame (≈10–15 m, drift-limited).
